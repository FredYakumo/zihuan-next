# Tool 使用限制机制

## 概述

Tool 使用限制机制的核心作用，是为 Agent 的能力调用建立边界。

它不仅仅是为了节省资源，也是在约束 Agent 的行为方式：哪些能力可以持续调用，哪些能力应该适可而止，哪些能力在达到上限后必须停止继续扩张。

从高层语义上看，这个机制表达的不是“某个功能能不能用”，而是“某种能力在当前上下文里还能被使用到什么程度”。

## 两层限制

这一机制由两层独立的约束组成。

### 单次会话限制

单次会话限制关注的是：在当前这轮对话里，同一个 Tool 最多可以被调用多少次。

它解决的是行为节奏问题。  
如果某个 Tool 的使用本来就应该是克制的，那么系统就不应该允许 Agent 在一次会话中反复尝试同一种能力，把一次回答过程拖成无休止的工具重试。

这一层的重点不在于长期配额，而在于保证当前对话过程保持收敛。

### 用户剩余额度限制

用户剩余额度限制关注的是：某个用户在更长时间范围内，对某种 Tool 还剩下多少可用次数。

它解决的是访问治理问题。  
也就是说，Tool 的使用不再被视为永远开放的能力，而是可以被当作一种受管理、可消耗的额度。

这一层表达的是长期边界，而不是单轮行为边界。

## 判断顺序

当 Agent 准备调用某个 Tool 时，系统会按固定顺序判断是否允许继续：

1. 先看当前会话里的调用次数是否已经达到上限。
2. 如果会话层面仍然允许，再看该用户是否还有剩余额度。
3. 只有两层都未命中限制时，Tool 才真正执行。

这个顺序体现的是两种不同的治理语义：

- 会话限制优先解决“这轮对话还能不能继续这样做”
- 用户额度限制再解决“这个用户整体上还是否拥有这项能力的使用余量”

## 命中限制后的语义

当限制被触发时，系统不会继续执行 Tool，也不会把这次限制伪装成一次成功调用。

相反，它会把一个限制结果返回给 Agent 自己，让 Agent 理解当前这项能力已经不可继续使用。

这意味着：

- Agent 应当把这条结果视为内部执行约束
- Agent 后续应停止继续调用这个已受限的 Tool
- Agent 不应直接把这段限制文本当作自然语言原样发给用户
- 只有当用户追问原因时，Agent 才应进一步解释限制的存在

这里的关键点是把“内部运行策略”与“面向用户的自然语言回复”明确分离开来
